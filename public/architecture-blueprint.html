Vue d'ensemble de l'écosystème
3 services · 1 base de données unifiée

FaîtiereHub est un écosystème à trois services partageant une seule base de données Supabase. Chaque service a un rôle distinct : FaîtiereHub est l'interface utilisateur, AgriTogo est le cerveau IA/ML, et Haroo est la couche de données des professionnels agricoles.

Chaîne de traitement d'une requête API

1
Vercel Edge — middleware.ts
Refresh session Supabase JWT → inject cookies → CSP + HSTS + X-Frame-Options headers
↓
2
Rate Limiting — lib/utils/rate-limit.ts
Vérification IP : in-memory (toujours) + Upstash Redis (si configuré). HTTP 429 si dépassé.
↓
3
Validation entrées — lib/validators/schemas.ts (Zod)
Schémas Zod stricts sur tous les paramètres. HTTP 400 avec message structuré si invalide.
↓
4
Assertion d'accès — lib/security/assert-access.ts
assertAuthenticated() → 401 si non-auth. assertRole() → 403 si rôle insuffisant. assertTenant() → 403 si hors périmètre.
↓
5
Logique métier + Supabase
Query PostgreSQL via client serveur. RLS appliquée automatiquement côté DB. Service role utilisé uniquement pour webhooks/admin.
↓
6
Headers de réponse — lib/security/headers.ts
Injection X-RateLimit-*, Cache-Control, Content-Type, security headers sur chaque réponse API.

Analyse marchés agricoles, prix, tendances. LLM : Gemini. Input : données prix Supabase.

Évaluation risque financier agriculteur. LLM : Claude (validation). Modèle GARCH volatilité.

Débat multi-modèles (Gemini vs Qwen). Décision consensus arbitrée par le Risk Agent.

Interface conversationnelle agriculteur. LLM : Qwen. Ton adapté aux coopératives.

Calcul besoins irrigation FAO-56. Multi-culture, bilan hydrique. Pas de LLM — calcul déterministe.

    Cas de Panne — Résilience "Et si ?"
    Dégradation gracieuse · Fallbacks · Recovery

    Matrice de résilience

    ⚠ Cas 1 — AgriTogo est hors ligne (Railway down)
    ImpactCartes OUVRIER / ACHETEUR / AGRONOME ne peuvent plus être vérifiées. Cartes FAITIERE non affectées.
    ComportementFaîtiereHub catch l'erreur fetch → retourne { source: 'error', message: 'Service temporairement indisponible' }
    RecoveryDès que Railway redémarre (auto-restart activé), les requêtes suivantes fonctionnent. Pas de données perdues.
    PréventionTimeout configurable (5s) sur fetch. Health check /api/v1/health. Railway restart policy = always.
    ⚠ Cas 2 — Supabase hors ligne
    ImpactToutes les pages authentifiées et vérifications QR échouent. Marketplace devient vide.
    ComportementErreurs 500 sur les API routes. L'Edge middleware ne peut pas refresh le JWT → redirect login.
    RecoverySupabase SLA 99.9%. Automatic failover Neon PG. Durée panne < 5 min historiquement.
    PréventionSentry capture toutes les erreurs DB. Upstash Redis garde les rate-limits même si Supabase down.
    ⚡ Cas 3 — Dépassement rate limit (attaque DDoS légère)
    ImpactEndpoint /api/verify/[card] : 10 req/min/IP. Au-delà : HTTP 429.
    ComportementIn-memory bucket décrémenté. Si Upstash configuré, compteur persistant cross-instance Vercel.
    RecoveryAutomatique — la fenêtre de 60s expire. L'utilisateur légitime peut relancer.
    PréventionHeaders X-RateLimit-Remaining retournés. Vercel WAF comme couche supplémentaire possible.
    ⚡ Cas 4 — Webhook KoboCollect avec signature invalide
    ImpactTentative injection de données terrain falsifiées.
    ComportementHMAC-SHA256 comparaison timing-safe échoue → HTTP 401 immédiat, corps vide. Aucune donnée traitée.
    RecoveryAucune action nécessaire. Les vraies soumissions KoboCollect ont la bonne signature.
    PréventionKOBO_WEBHOOK_SECRET rotatif. Audit log de toutes les tentatives invalides.
    ✓ Cas 5 — Open-Meteo API lente ou timeout
    ImpactModule météo lent ou incomplet sur la page de vérification.
    ComportementLes 6 sources sont fetched en Promise.all. Si 1 ou 2 sources timeout → fusion partielle avec les modèles disponibles. Si cache Supabase < 3 jours → retourne cache même incomplet.
    RecoveryCache in-memory par région (TTL 15–30 min) absorbe les piques. Prochaine requête régénère.
    PréventionOpen-Meteo free tier — SLA non garanti. Cache agressif dimensionné pour 30 min sans appel externe.
    ✓ Cas 6 — Build Next.js sans variables d'environnement (.env local manquant)
    ImpactSur Vercel prod : impossible — vars intégrées au build time. En dev local : NEXT_PUBLIC_* absentes.
    Comportementlib/supabase/client.ts détecte typeof window === 'undefined' + vars absentes → retourne client placeholder SSR-safe. Pas de crash de prerendering.
    RecoveryAjouter .env.local avec les vraies vars. Le placeholder n'est jamais utilisé en production.
    PréventionLe script npm run security:check vérifie la présence des vars avant déploiement.
